# Modular CI/CD Pipeline Architecture — Project-Independent Specification

**Objective:** Create a universal, project-agnostic CI/CD pipeline system that supports ANY project (language, framework, service count) deploying to ANY platform (EKS, ECS, Azure Container Apps, etc.) using ANY registry (ECR, ACR, Docker Hub).

---

## 🎯 Core Principle

**PIPELINES SHOULD BE PROJECT-INDEPENDENT**

The system must work identically for:
- Node.js backend + React frontend
- Python data pipeline
- Go microservice
- Java Spring Boot app
- Multi-language monorepo
- Single service microservice
- 100-service architecture

Without code changes to pipeline files — only configuration changes.

---

## 🏗️ Three-Layer Architecture

```
┌─────────────────────────────────────────────────────────┐
│ LAYER 1: PIPELINE GENERATION AGENT                      │
│ Input: "I want to deploy my NodeJS app to EKS with ECR" │
│ Output: Customized caller workflows                      │
└─────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────┐
│ LAYER 2: CALLER WORKFLOWS (PROJECT-SPECIFIC)            │
│ Location: Any project's .github/workflows/              │
│ Files: ci-build.yml, cd-deploy.yml (thin, minimal)      │
│ Role: Choose platform, registry, services to build      │
└─────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────┐
│ LAYER 3: REUSABLE WORKFLOWS (UNIVERSAL LIBRARY)         │
│ Location: Shared org repo (.github or templates)        │
│ Files: ci-build-reusable.yml, cd-deploy-reusable.yml    │
│ Role: Execute deployment logic for ANY project          │
│ Supports: All languages, all platforms, all registries  │
└─────────────────────────────────────────────────────────┘
```

## 📋 LAYER 1: Pipeline Generation Agent

### Input Interface (Project-Agnostic)

```yaml
# User provides their project requirements
pipeline_config:
  # Project basics
  project_name: "any-app"
  repository_url: "https://github.com/org/any-app"
  
  # Build configuration
  build:
    registry: "ECR" | "ACR" | "DockerHub"
    registry_url: "123456.dkr.ecr.us-east-1.amazonaws.com"
    services:
      - name: "service-name"
        build_context: "path/to/service"  # Relative path in repo
        dockerfile: "Dockerfile"
        image_name: "service-name-image"
        # Language/framework auto-detected from Dockerfile
  
  # Deployment configuration
  deploy:
    platform: "EKS" | "ECS" | "AzureContainerApps" | "CloudRun"
    
    # Platform-specific (only relevant ones required)
    eks:
      cluster_name: "my-cluster"
      region: "us-east-1"
      namespace: "default"
    
    ecs:
      cluster_name: "my-cluster"
      region: "us-east-1"
      service_name: "my-service"
    
    azure:
      resource_group: "my-rg"
      container_app_name: "my-app"
      environment_name: "my-env"
```

### Agent Output

```
Generated Files:
  ✓ ci-build.yml
  ✓ cd-deploy.yml
  ✓ Optional: k8s/base/deployment.yaml (if EKS)
  ✓ Optional: ecs-task-definition.json (if ECS)
  ✓ SETUP.md (deployment instructions)
```

### Agent Behavior (Project-Independent)

```
1. Parse configuration
2. Validate platform + registry combination
3. Detect project type from Dockerfile/package.json/go.mod/etc.
4. Generate caller files (ci-build.yml, cd-deploy.yml)
   - References shared reusable workflows
   - Passes all config as parameters
5. Generate platform-specific manifests (if needed)
6. Output git-ready files
7. Provide deployment checklist
```

## 📋 LAYER 2: Caller Workflows (Project-Specific)

### ci-build.yml — Universal Pattern, Project-Specific Config

```yaml
name: CI Build
on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main, develop]

jobs:
  ci-build:
    uses: .github/.github/workflows/ci-build-reusable.yml@main
    # OR: uses: org-repo/.github/workflows/ci-build-reusable.yml@main
    
    with:
      # These are GENERATED by agent, specific to this project
      build_registry: ${{ vars.BUILD_REGISTRY }}
      registry_url: ${{ secrets.REGISTRY_URL }}
      registry_region: ${{ vars.REGISTRY_REGION }}
      
      services: |
        [
          {
            "name": "backend",
            "build_context": "backend",
            "dockerfile": "Dockerfile",
            "image_name": "my-backend"
          },
          {
            "name": "frontend",
            "build_context": "frontend",
            "dockerfile": "Dockerfile",
            "image_name": "my-frontend"
          }
        ]
    
    secrets: inherit
```

### cd-deploy.yml — Universal Pattern, Project-Specific Config

```yaml
name: CD Deploy
on:
  workflow_run:
    workflows: [CI Build]
    types: [completed]
    branches: [main, develop]
  workflow_dispatch: {}

jobs:
  deploy:
    uses: .github/.github/workflows/cd-deploy-reusable.yml@main
    
    with:
      # These are GENERATED by agent, specific to this project
      deployment_platform: ${{ vars.DEPLOYMENT_PLATFORM }}
      build_registry: ${{ vars.BUILD_REGISTRY }}
      registry_url: ${{ secrets.REGISTRY_URL }}
      image_sha: ${{ github.event.workflow_run.head_sha || github.sha }}
      
      # Platform-specific (only relevant ones used)
      eks_cluster_name: ${{ vars.EKS_CLUSTER_NAME }}
      eks_region: ${{ vars.EKS_REGION }}
      eks_namespace: ${{ vars.EKS_NAMESPACE }}
      
      ecs_cluster_name: ${{ vars.ECS_CLUSTER_NAME }}
      ecs_region: ${{ vars.ECS_REGION }}
      ecs_service_name: ${{ vars.ECS_SERVICE_NAME }}
      
      azure_resource_group: ${{ vars.AZURE_RESOURCE_GROUP }}
      azure_container_app_name: ${{ vars.AZURE_CONTAINER_APP_NAME }}
    
    secrets: inherit
```

### Key: These Workflows Are THIN

```
What they DON'T do:
  ❌ No build logic
  ❌ No deployment logic
  ❌ No platform-specific code
  ❌ No language-specific commands

What they DO do:
  ✅ Choose which shared reusable to call
  ✅ Pass project config
  ✅ Nothing else
```

## 📋 LAYER 3: Reusable Workflows (Universal Library)

### ci-build-reusable.yml — Language/Project Agnostic

**Inputs (Fully Generic):**
```yaml
inputs:
  build_registry:
    type: string
    description: "ECR, ACR, or DockerHub"
  
  registry_url:
    type: string
    description: "registry.example.com"
  
  services:
    type: string
    description: "JSON array of {name, build_context, dockerfile, image_name}"
  
  registry_region:
    type: string
    description: "AWS region (for ECR only)"
```

**Logic (Fully Generic):**
```yaml
jobs:
  detect-changes:
    # Read from git; determine which services changed
    # Project-independent: works for any structure
  
  build:
    strategy:
      matrix:
        service: ${{ fromJSON(inputs.services) }}
    steps:
      # Generic Docker build from any Dockerfile
      - docker/build-push-action
        with:
          context: ${{ matrix.service.build_context }}
          file: ${{ matrix.service.dockerfile }}
          tags: local/${{ matrix.service.name }}:$SHA
  
  scan:
    # Generic Trivy scan (language-independent)
  
  push-ecr:
    if: ${{ inputs.build_registry == 'ECR' }}
    # ECR-specific logic
  
  push-acr:
    if: ${{ inputs.build_registry == 'ACR' }}
    # ACR-specific logic
  
  push-dockerhub:
    if: ${{ inputs.build_registry == 'DockerHub' }}
    # DockerHub-specific logic
```

**Key Properties:**
- ✅ Works for Node.js, Python, Go, Java, etc. (just needs Dockerfile)
- ✅ Works for single service or 100 services (matrix-based)
- ✅ Registry-agnostic (conditional jobs)
- ✅ No language detection needed

### cd-deploy-reusable.yml — Platform Agnostic

**Inputs (Fully Generic):**
```yaml
inputs:
  deployment_platform:
    type: string
    description: "EKS, ECS, AzureContainerApps, CloudRun, etc."
  
  image_sha:
    type: string
    description: "Commit SHA for image tag"
  
  # Platform-specific (agent passes only relevant ones)
  eks_cluster_name:
    type: string
  ecs_cluster_name:
    type: string
  azure_resource_group:
    type: string
  # ... etc
```

**Logic (Platform-Specific Conditional Jobs):**
```yaml
jobs:
  deploy-eks:
    if: ${{ inputs.deployment_platform == 'EKS' }}
    steps:
      - OIDC auth
      - kubectl setup
      - kustomize + kubectl apply
      - Rollout wait
      - Health check
  
  deploy-ecs:
    if: ${{ inputs.deployment_platform == 'ECS' }}
    steps:
      - OIDC auth
      - Update task definition
      - Update service
      - Wait for stabilization
      - Health check
  
  deploy-azure:
    if: ${{ inputs.deployment_platform == 'AzureContainerApps' }}
    steps:
      - Azure auth
      - Update container app
      - Wait for deployment
      - Health check
  
  deploy-cloudrun:
    if: ${{ inputs.deployment_platform == 'CloudRun' }}
    steps:
      - GCP auth
      - Deploy to Cloud Run
      - Health check
```

**Key Properties:**
- ✅ Platform-agnostic caller (doesn't know which platform)
- ✅ Conditional jobs (only relevant platform runs)
- ✅ Generic patterns (all use OIDC, all have health checks)
- ✅ Supports new platforms by adding conditional job

## 🤖 Pipeline Generation Agent — Algorithm

```
INPUT: User provides pipeline_config.yaml

STEP 1: PARSE
  Extract:
    - Project name
    - Services list
    - Build registry
    - Deployment platform
    - Platform-specific parameters

STEP 2: VALIDATE
  Verify:
    - Platform exists in reusable workflows
    - Registry exists in reusable workflows
    - Services have valid build_context + dockerfile
    - Platform parameters are valid

STEP 3: GENERATE CALLER FILES
  Create ci-build.yml:
    - uses: shared-reusable-build-workflow
    - with: (build_registry, registry_url, services, region)
  
  Create cd-deploy.yml:
    - uses: shared-reusable-deploy-workflow
    - with: (deployment_platform, platform-specific-params)

STEP 4: GENERATE OPTIONAL MANIFESTS
  If EKS:
    - Generate k8s/base/deployment.yaml
    - Generate k8s/base/service.yaml
    - Generate k8s/overlays/prod/kustomization.yaml
  
  If ECS:
    - Generate ecs-task-definition.json
  
  If Azure:
    - Generate azure-container-app.yaml

STEP 5: OUTPUT
  Git-ready files:
    ✓ .github/workflows/ci-build.yml
    ✓ .github/workflows/cd-deploy.yml
    ✓ Optional manifests
    ✓ SETUP.md (instructions)

STEP 6: VERIFY
  Confirm:
    - Files reference correct reusable workflows
    - All parameters will be resolved
    - No missing repository variables/secrets
```

## ✅ Key Principles (PROJECT-INDEPENDENT)

1. **No Language Assumptions**
   - Reusable workflows accept any Dockerfile
   - No Node.js-specific npm commands
   - No Python-specific pip commands
   - Works for compiled and interpreted languages

2. **No Service Count Assumptions**
   - Matrix-based (works for 1 service or 100)
   - Change detection is generic
   - Build parallelization is automatic

3. **No Platform Assumptions**
   - Conditional jobs (only relevant platform runs)
   - Each platform is self-contained
   - Adding new platform = add new conditional job

4. **No Registry Assumptions**
   - Conditional registry jobs
   - Generic auth pattern (OIDC)
   - Works with ECR, ACR, Docker Hub, etc.

5. **Single Source of Truth**
   - Reusable workflows maintained once
   - All projects benefit from improvements
   - No duplication across repos
